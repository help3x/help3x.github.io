文章

仕様を確かめる・決める

  1. 読者

    必ず設定する。

    文章は読み手のために書くものである。

    読み手が決まれば、文体や言葉の難易度はおのずと決まる。

  2. 目的

    何のために書くのかを明確にする。

    読者を楽しませるのか、自分の主張を伝えて読者の行動を変えるのか、など。

  3. 分量

    字数を制限する。 手持ちの素材をすべて詰め込むより、優先順位をつけて絞り込んだほうが、文章が引き締まる。

  4. 締め切り

    もし締め切りがなくても、自分の締め切りを設定するクセをつける。

スケルトンを作る

仕様が固まったら、それに従い文章の設計図（スケルトン）を作る。

  1. 仮見出し（15～25字）

  2. 要約、要旨（150～200字）

  3. 本文の構成（1段落文で1行）

原稿を書くための見出しと読者に読ませる見出しは違うもの。 ここで立てる「仮見出し」というのは、原稿を書くための見出しのこと。 これには記事に含まれている5W1Hのうち、最も重要な要素を盛り込む。 「結論」や「主張」が一目でわかる見出しを立てる。

「～について」というテーマのような見出しではいけない。

基本4パターン

  1. 逆三角形（伝える）
  2. 三部構成（論じる）
  3. 起承転結（心を動かす）
  4. 起承展転結（心を動かす）

読みやすい文章の三原則

  1. 一文は40字、長くても60字におさめる。

  2. 一文一意の原則。

  3. 能動態だけで書く。受動態は原則禁止。

    「～された」という表現は、行為の主体が曖昧になりやすいため。 たとえば、「○月×日に運動会が開かれた」という表現は文法として間違ってはいないが、「誰が」という情報が抜け落ちている。

React-Static Introduction

Powered by Google 翻訳。

Dev Stage

  1. サイトがレンダリングする必要があるすべてのデータは、必要な手段を使用して設定ファイルにまとめられます。このデータは、Markdownファイル、ヘッドレスCMS、Graphqlエンドポイント、パブリックAPIのどこからでも来ることができます。

  2. 次に、サイトのすべてのルートのリストを生成し、それをレンダリングするためのテンプレートを定義し、それぞれに適切なデータを提供します。props を渡すのと同じくらい簡単です！

  3. RoutePropsやSitePropsのような React-Static のコンポーネントを使用すると、通常どおりルートごとにデータにアクセスしてアプリを書くことができます。

  4. React-Staticは、サイト内のすべてのページを高速かつ正確にエクスポートできます。

  5. お気に入りの静的サイトホストまたは CDN にサイトをアップロードしてください！

Client Stage

  1. ユーザーの最初のページで、静的にエクスポートされた HTML と CSS がダウンロードされ、できるだけ早くコンテンツが表示されます。

  2. 同時に JS の最低限のものが、レンダリングされた HTML の上に React to bootstrap のためにダウンロードされます。

  3. できるだけ早く、ウェブサイトの残りの部分は楽観的にプリフェッチされ、コード分割とデータ分割によってキャッシュされます。

  4. それ以上のナビゲーションはすべて即時です！

開発時に意味すること

  • ライフサイクルセキュリティ、ルートアニメーションなどこのフードでは、react-staticは react-router v4を使用し、カスタムライフサイクルメソッドを決して使用しません。つまり、dangerouslySetInnerHTML や他の破壊メソッドがナビゲーションに使用されていません。 それはちょうどいい古い React なので、コンポーネントのライフサイクルや state の管理者など、あなたが通常期待していることをすべて機能していることに徹底的に頼ることができます。

  • データとクエリの不可知論。 Webサイトのすべてのデータを作成して整理するだけで十分な作業をしたので、データとページの間に余計なGraphQLレイヤーやカスタムコンポーネントのライフサイクルを置く必要があります。 react-staticでは、あなたのルートにデータを供給するのが簡単です。 必要なものを取り出し、ルートに一度にデータを渡してください。 さまざまなreact-staticコンポーネントを使用して、サイト内のどこでもこのデータを使用することができます（render props、HOC’s、私たちはすべてそれを持っています）。 データの取り込み、クエリー、マッピングのすべてを単一の場所に移動することで、データパイプラインのコードを3倍少なくして、データの形状を管理する中心的な場所を確保できます。

  • 今から2年後には、Leanが構築されています。 react-staticのデータ分割手法は非常にユニークで、すべてのルートのテンプレートとデータが最適化され、マテリアライズされた分散APIとして提供されることを保証します。 つまり、サイトを数万ページに拡張しても、即座に読み込むことができます。 繰り返しますが、すべてのデータフェッチが同じ単一ステージで実行されるため、 react-staticのビルド時間は信じられないほど速いです。

結論

自前で作り込む内容が多い。 たとえば、Markdown および front-matter の読み込み。

経路はstatic.config.jsで定義できるが、その前に読み込みと加工が終わっていなければならない。

お膳立てしてあげなければいけないことが、思っていたより多いかもしれない。

大元の内容を Markdown で書けば、あとはいいようにやってくれるというツールではなかった。

Material UI のデモページのメモ。

  • ダイアログコンポーネントは state を持たない。初期値や閉じる処理は props でもらう。

  • 呼び出し元の state にダイアログのオープン／クローズを持つ。

  • ダイアログのクローズ処理を props で渡す。

  • Dialog の props.open でダイアログの表示／非表示を制御できる。

自作した AlertDialog コンポーネントの話。

props

  • open

    ダイアログを開くのであれば true を、閉じるのであれば false を指定する。

  • onClose

    ダイアログを閉じたときの処理を指定する。

  • onYesClick

    OK ボタンをクリックしたときの処理を指定する。

  • title

    タイトルを表示したければ指定する（任意）。

  • messages

    表示する文字列を配列で指定する。

    配列内の値が falsy であれば無視される。ただし、空文字は空行として表示される。\nは改行に変換される。\\nは改行ではなく\nと表示される。

    最大表示数を超えた分は表示されず、「上記のほか、n 件のメッセージがあります。」になる。

CMS.js

Node.js や PHP 、Ruby といったサーバーサイドスクリプトは使わない。

IE11 でデモページを見たが動かなかった。 Polyfill を入れる必要がある。

オブジェクトは 'assign' プロパティまたはメソッドをサポートしていません。

Link to GitHub –> chrisdiana/cms.js

動作モードには次の2つがある。

  • Github モード

    GitHub API を使ってリポジトリの内容を取得し、サイトを構築する。

  • Server モード

    サーバーから取得したディレクトリインデックスを使って、サイトを構築する。
    Markdown または HTML ファイルを対象とする。

    個人的には、セキュリティを考慮するとあまり使いたくないモード。

サニタイジング

テキスト

HTML のタグではない部分に入力値を埋め込む場合、次の3文字を置き換える。

  • & –> &amp;
  • < –> &lt;
  • > –> &gt;

タグ属性値

HTML タグの属性値に入力値を埋め込む場合、次の5つの文字を置き換える。 そして置き換えた文字をダブルクォート（またはシングルクォート）で囲み、属性値に指定する。

  • & –> &amp;
  • < –> &lt;
  • > –> &gt;
  • " –> &quot;
  • ' –> &#39;

属性値は必ずダブルクォートまたはシングルクォートでくくること！

これは安全。

<img src="$selected_icon">

これは危険。

<img src=$selected_icon>

もし$selected_iconの値がno_such_icon onerror=alert(document.cookie);であると、次のように展開されてしまう。

<img src=no_such_icon onerror=alert(document.cookie);>

Pandoc パイプテーブルのメモ

  • パイプテーブルのセルには、段落やリストなどのブロック要素を含められない。また、複数の行にまたがることもできない。

  • パイプテーブルに列の幅よりも広い行が含まれている場合、テーブルは全文の幅を占める。セルの内容は折り返され、相対的なセルの幅は、表の見出しと表の本文を区切る行にダッシュが表示される（どういうこと？）。

    たとえば、---|-とした場合、最初の列の幅は 3/4 、二番目の列の幅は 1/4 になります。

    列の幅は --columns オプションで指定できる。規定値は 72 。

    符号なし 16 ビット整数の最大値は指定できた ⇒ --columns=65536

    符号付き 32 ビット整数の最大値も指定できた ⇒ --columns=2147483647

    符号なし 32 ビット整数の最大値は指定できなかった ⇒ --columns=4294967295としたらcolumns must be a number greater than 0が出た。「列は 0 より大きい数値でなければなりません」とのこと。

    この時点でアウト。符号付き 32 ビット整数の最大値＋１ ⇒ --columns=2147483648　エラーの内容は同じ。

    ということで、最大の幅を指定したいときは符号付き 32 ビット整数の最大値（--columns=2147483647）を指定する。

  • 列の幅よりも幅の広い行がない場合、セルの内容は折り返されない。セルの幅は内容の長さに合わせられる。

ログインユーザーへの権限付与

ログイン時に編集可否を判定する。

編集可否の判定は、その時点のログイン情報をもとにする。 もし、ユーザーのログイン情報が残っていても、それは考慮しない。 （ログイン情報が残る＝ログアウトせずにブラウザ・タブを閉じた場合）

例1:
Ａさんがログインして編集権限を付与された。 そのあと、Ｂさんがログインした。Ａさんが先にログインしているため、参照権限が付与された。 Ａさんはログアウトせずにブラウザを閉じた。 Ａさんがもう一度ログインした。Ａさんより先にＢさんがログインしているため、参照権限が付与された。

例2:
Ａさんがログインして編集権限を付与された。 Ａさんはログアウトせずにブラウザを閉じた。 Ａさんがもう一度ログインした。Ａさんより先にログインしているユーザーはいないため、参照権限を付与された。
